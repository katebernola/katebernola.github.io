# 01 · Philosophy

## Purpose

The core stance everything else builds on. If a rule elsewhere conflicts with this book, this book wins — or the conflict gets logged in `thinking/we-might-be-wrong.md`.

## Principles

- The copy is the last decision, not the first. Everything before it is why the words hold up.
- Communication is a decision process, not a writing task. Most of the work is deciding what *not* to say.
- Words cannot fix a design problem. They can only narrate it. Ask first: if the design changed, would this message still be needed?
- Serve the user's goal in the moment. The business goal gets served by the trust that builds — or it gets its own, better moment.
- Every message spends attention. Attention is the budget the lead message depends on.

## Rules

- Never write copy before the strategy exists: journey stage, user goal, business goal, communication goal, message verdicts.
- Never simply accept a request's assumptions. Surface them and make the requester respond.
- Present strategy before copy in reviews. Invite disagreement at the decision level — it is cheaper there.

## Borrowed experience values

Six lenses for critiquing any experience, adopted from Polaris until internal values land *(source: Polaris experience values)*:

- **Considerate** — don't assume the perfect situation, preparation, or usage.
- **Empowering** — help users complete tasks without taking away decisions or independence; hide complexity but keep sophistication reachable.
- **Crafted** — delight with how effective the experience is, not with extra decoration.
- **Efficient** — use friction to highlight important decisions, then let users get back to work.
- **Trustworthy** — be transparent about what tools can and can't do; make safe actions frictionless and risky ones clearly instructed and recoverable; fix the small inconsistencies that erode trust.
- **Familiar** — use recognized patterns; evolve experiences seamlessly.

## Decision heuristics

- If you can't state the problem behind the ask in one sentence, go ask before writing a word.
- If a stakeholder pushes to keep a message, ask what the user would do differently after reading it.
- Nice to know is not need to know.

## Common mistakes

- Treating a stakeholder's proposed solution ("add a banner") as the problem. See `../mistakes/business-vs-user-goals.md`.
- Writing to cover every case, which serves no case. See `../mistakes/too-many-options.md`.
